1. Visão Geral do Sistema

Objetivo: Fornecer ao setor de compras do Empório Villa Borghese uma sugestão precisa de quantidade e valor de pedido por fornecedor, já considerando: custo real fiscal (ICMS, PIS/COFINS, ST), margem líquida mínima de 30–35%, histórico de vendas, estoque atual, quebras e rupturas de estoque, transferências entre unidades e projeção de pagamento versus calendário de contas a pagar — tudo conectado em tempo real ao ERP Hipcom.

O sistema é composto pelos seguintes arquivos:

ArquivoFunção
proxy.jsServidor Node.js puro. Conecta na API Hipcom, processa cálculos fiscais e de sugestão de compra (incluindo multi-loja e transferências), gerencia banco SQLite local, expõe API REST na porta 3000.
index.htmlInterface web completa (HTML + CSS + JS puro). Duas abas: "Análise de Compras" e "Fluxo de Caixa". Acesso via navegador em http://localhost:3000.
db.jsCamada de acesso ao SQLite (banco.db). Gerencia cache local de dados Hipcom, contas a pagar e prazos por fornecedor.
banco.dbBanco SQLite gerado automaticamente. Persiste vendas, estoques, compras, fiscal, contas a pagar e prazos — evita repetir chamadas ao Hipcom.

5. Cálculos Fiscais — Lucro Real / SP

5.1 Custo Real (após recuperação de créditos na entrada)

custo_real = custo_hipcom - (custo_hipcom × icmsAlqEntrada / 100) - (custo_hipcom × pisAlqEntrada / 100) - (custo_hipcom × cofinsAlqEntrada / 100) Obs.: No regime Lucro Real, a empresa recupera crédito de ICMS, PIS e COFINS nas compras. O custo_real é o custo efetivo após esses créditos.

5.2 Impostos na Saída (débitos ao vender)

deb_icms = preco_venda × icmsAlqSaida / 100 (= 0 se produto com ST — CST saída 60, imposto já pago na entrada) deb_pis = preco_venda × pisAlqSaida / 100 deb_cofins = preco_venda × cofinsAlqSaida / 100 deb_fcp = preco_venda × fcp / 100 (Fundo de Combate à Pobreza) total_impostos_saida = deb_icms + deb_pis + deb_cofins + deb_fcp

5.3 Margem Líquida Real

margem_bruta = (preco_venda - custo_hipcom) / preco_venda × 100 margem_liquida = (preco_venda - custo_real - total_impostos_saida) / preco_venda × 100 Meta: 30% ≤ margem_liquida ≤ 35%

5.4 Preço Mínimo de Venda (sem prejuízo)

aliquota_total_saida = (icmsAlqSaida + pisAlqSaida + cofinsAlqSaida + fcp) / 100 // Para ST ativa (CST 60): icmsAlqSaida = 0 no denominador preco_minimo_30% = custo_real / (1 - aliquota_total_saida - 0.30) preco_minimo_35% = custo_real / (1 - aliquota_total_saida - 0.35)

5.5 Regimes PIS/COFINS detectados automaticamente

RegimeComo é detectadoImpacto no cálculo
Normal Alíquotas PIS/COFINS > 0 nos campos de entrada e saída do Hipcom Crédito na entrada + débito na saída (regime não-cumulativo)
Monofásico Alíquotas todas zero; CST saída 07, 08 ou 09 (bebidas, combustíveis, etc.) Sem crédito na entrada, sem débito na saída. Imposto recolhido pela indústria.
Isento Alíquotas todas zero; CST saída 04, 05 ou 06 (cesta básica, Lei 10.925/2004) Alíquota zero. Carnes, laticínios, grãos, hortifrutigranjeiros.

6. Substituição Tributária (ST) e Alertas Fiscais

O sistema cruza os dados fiscais reais do Hipcom com a tabela NCM local para detectar situações que impactam a margem:

AlertaCausaO que fazer
ST_REMOVIDA O NCM do produto teve a Substituição Tributária extinta (ex: azeite — Portaria SRE 14/2024; limpeza — Decreto 67.301/2022). O Hipcom ainda pode ter o produto cadastrado com CST 60. Verificar se o fornecedor está emitindo NF sem ST. Se sim, atualizar cadastro no Hipcom e revisar o preço de venda (margem melhora).
DIVERGENCIA_CST O CST cadastrado no Hipcom diverge do regime esperado para o NCM em SP. Conferir NF de entrada e cadastro fiscal no Hipcom. Solicitar validação ao contador.
MONOFASICO Produto com PIS/COFINS monofásico (água, refrigerante, cerveja, combustíveis). Não há crédito de PIS/COFINS na entrada. Margem calculada sem esses créditos.
ISENTO Produto isento de PIS/COFINS (cesta básica, carnes, laticínios). Alíquota zero. Impacto positivo na margem — confirmar isenção no cadastro Hipcom.
SEM_NCM Produto sem NCM cadastrado na Hipcom. Cadastrar o NCM correto no Hipcom para que o sistema calcule o regime adequado.
SEM_FISCAL Endpoint fiscal do Hipcom não retornou dados para o produto. Cálculo estimado pela tabela NCM. Verificar cadastro fiscal no Hipcom.

7. Sugestão de Quantidade de Compra

7.1 Fórmula completa (multi-loja)

// Dados agregados de TODAS as lojas selecionadas venda_media_dia = Σ(total_vendido_por_loja) / dias_periodo estoque_atual = Σ(estoque_atual_por_loja) total_compras = Σ(compras_por_loja) // Quebra real (perda de estoque) — calculada sobre agregado quebra_estimada = (estoque_inicio + total_comprado) - total_vendido - estoque_fim taxa_quebra (%) = quebra_estimada / (estoque_inicio + total_comprado) × 100 // Ruptura: % de dias sem venda quando produto tem giro ruptura_pct (%) = (dias_sem_venda / dias_periodo) × 100 (somente se venda_media > 0) // Quantidade necessária para cobrir X dias de abastecimento qtd_necessaria = venda_media_dia × dias_abastecimento qtd_com_quebra = qtd_necessaria × (1 + taxa_quebra / 100) qtd_liquida = qtd_com_quebra - estoque_atual // Ajuste por ruptura frequente (>30% dos dias sem venda = falha de estoque) se ruptura_pct > 30%: qtd_liquida = qtd_liquida × 1.20 (+20% de segurança) // Arredondar para múltiplo da embalagem qtd_sugerida = MAX(0, qtd_liquida) qtd_sugerida = TETO(qtd_sugerida / qtd_embalagem) × qtd_embalagem // ── Transferência entre lojas (antes de comprar do fornecedor) ──────────── // Para cada loja com excesso (estoque > necessidade × 1.3): excesso_loja = MAX(0, estoque_loja - necessidade_loja × 1.3) qtd_transf_loja = PISO(excesso_loja / qtd_embalagem) × qtd_embalagem qtd_transferencia = Σ(qtd_transf_loja de todas as lojas com excesso) [≤ qtd_sugerida] qtd_comprar = MAX(0, qtd_sugerida - qtd_transferencia) // Valor do pedido (apenas o que vai ser comprado de fato) valor_pedido = qtd_comprar × custo_hipcom
Fluxo de decisão de compra: O sistema primeiro verifica se há excesso em alguma loja da rede. Se sim, propõe a transferência interna (mais barato e rápido). Só o saldo restante (qtd_comprar) vai para o pedido ao fornecedor. Exemplo: precisa de 48 unidades, tem 24 para transferir → pede apenas 24 ao fornecedor.

11. Estrutura da Resposta por Produto

Cada produto retornado pela análise possui os seguintes campos:

📦 Identificação

  • plu — código do produto no Hipcom
  • nome — descrição do produto
  • ncm — código NCM (8 dígitos)
  • departamento / secao
  • curva_abc — A / B / C

📊 Movimentação

  • estoque_atual — unidades em estoque hoje
  • estoque_inicio — estoque no início do período
  • total_vendas_periodo — total vendido (unid.)
  • total_compras_periodo — total comprado (unid.)
  • venda_media_dia — média diária de venda
  • taxa_quebra (%) — perda estimada no período
  • ruptura_pct (%) — % dias sem venda com giro

🛒 Sugestão de Compra

  • qtd_embalagem — múltiplo obrigatório do pedido
  • qtd_sugerida — necessidade total (agregada de todas as lojas)
  • qtd_transferencia — pode ser suprida por excesso em outra loja
  • qtd_comprar — qtd real a pedir ao fornecedor (sugerida − transferência)
  • valor_pedido — R$ custo × qtd_comprar
  • receita_prevista — R$ se vender tudo no período
  • lucro_previsto — R$ lucro líquido projetado
  • recomenda_compra — true / false
  • status — OK / REVISAR PREÇO / etc.
  • transferencias[] — array {de_loja, qtd, excesso} para cada loja doadora
  • por_loja[] — breakdown individual {loja, estoque, venda_media_dia, excesso}
  • lojas_analise[] — IDs das lojas incluídas na análise

💰 Fiscal

  • custo_hipcom — custo bruto do Hipcom
  • custo_real — custo após créditos de entrada
  • margem_bruta (%) — sem fiscal
  • margem_liquida (%) — com todos impostos
  • preco_min_30 — preço mínimo para 30% ML
  • preco_min_35 — preço mínimo para 35% ML
  • tem_st — true se CST saída = 60
  • regime_pis_cofins — normal / monofasico / isento
  • cenarios_reforma — projeção 2025→2033

15. Importação TXT — Contas a Pagar (Hipcom)

O módulo Contas a Pagar do Hipcom não possui endpoint REST na versão v7.3. A solução adotada é a importação quinzenal do relatório "Calendário dos Pagamentos a Efetuar" exportado como TXT.

Como exportar do Hipcom: Financeiro → Contas a Pagar → Calendário dos Pagamentos a Efetuar → Selecionar período (ex: 01/06/2026 a 30/06/2026) → Imprimir como TXT. Frequência recomendada: a cada 15 dias (quinzenal).

15.1 Formato do arquivo TXT

CaracterísticaDetalhe
CodificaçãoWindows-1252 (Latin-1) — lido com FileReader.readAsText(file, 'windows-1252')
Separador de colunasPipe | — 8 colunas (Sáb / Dom / Seg / Ter / Qua / Qui / Sex / Total semana)
Linha de diasContém padrão (dd) — ex: (15) para o dia 15
Linha LLançamentos financeiros confirmados — ex: L 187.060,58
Linha PPrevisões financeiras — ex: P 12.500,00
Cabeçalho do mêsMês por extenso em letras espaçadas — ex: J U N H O / 2 0 2 6
Lojas inclusasIndicadas no cabeçalho — ex: Lojas:1,2,6

15.2 Parser TXT (parseTxtHipcom)

// 1. Detecta mês/ano pelo padrão de letras espaçadas // "J U N H O / 2 0 2 6" → { mes: 6, ano: 2026 } // 2. Detecta lojas pela linha "Lojas:1,2,6" // 3. Para cada linha com |(dd)| detecta o dia do mês // Lê as próximas linhas: primeira com |L = lançamentos, primeira com |P = previsões // Divide pelo separador | → 7 colunas de dias da semana // 4. Converte valor BR → float // "2.789,60" → 2789.60 (remove pontos, substitui vírgula) // 5. Resultado por dia: { data: "2026-06-15", valor_lancamento, valor_previsao } // Validação: total de lançamentos do mês deve bater com "LANÇAMENTOS DO MÊS" no rodapé
